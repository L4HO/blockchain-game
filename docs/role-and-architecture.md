# 블록체인 연동 및 테스트 환경 구성 담당 범위

## 담당 목표

- Geth 로컬 네트워크 실행
- 스마트 컨트랙트 배포
- 모바일 신분증 검증 결과를 온체인 접근 권한으로 기록
- 게임 플레이 결과 트랜잭션 기록
- 트랜잭션 해시 기반 검증 및 테스트

## 온체인에 저장하는 데이터

개인정보 원문은 저장하지 않는다.

- `mobileIdHash`: 모바일 신분증 검증 결과에서 만든 해시
- `player`: 플레이어 지갑 주소
- `sessionId`: 게임 플레이 1회에 대한 고유 해시
- `gameId`: 게임 종류 해시
- `score`: 점수
- `playedAt`: 기록 시각

## 인증 흐름

1. 사용자가 모바일 신분증으로 인증한다.
2. 백엔드가 모바일 신분증 사업자 또는 DID 검증 API에 토큰을 검증한다.
3. 백엔드는 검증된 사용자 식별값을 해시로 변환한다.
4. `GameAccessLog.grantAccess()` 트랜잭션으로 접근 권한을 기록한다.
5. 게임이 끝나면 `GameAccessLog.recordGame()` 트랜잭션으로 결과를 기록한다.
6. 제출용 검증은 트랜잭션 해시, 블록 번호, 이벤트 로그를 확인한다.

## 수업 시연 순서

1. `npm run geth:start`
2. 새 터미널에서 `npm run deploy:geth`
3. `npm run record:sample`
4. 출력된 트랜잭션 해시로 `npm run verify:tx -- <txHash>`
5. `npm test`

## 실제 모바일 신분증 연동 시 교체 지점

`src/auth/mobile-id.ts`의 `MockMobileIdVerifier`를 실제 검증 API 클라이언트로 교체한다.

예상 구현 방식:

- 앱/프론트가 모바일 신분증 인증 토큰을 백엔드로 전달
- 백엔드가 사업자 API에서 서명, 만료 시간, 성인 여부, 필요한 속성을 검증
- 원본 속성은 저장하지 않고 `ethers.id()` 또는 서버 측 salt를 포함한 해시만 컨트랙트에 전달
